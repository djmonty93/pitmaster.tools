# MailerLite → Sender.net Migration — Design

**Date:** 2026-05-16
**Status:** Approved (pending writing-plans handoff)
**Approach:** Mirror replace with Sender-shaped client (Option A)
**Scope:** Greenfield reset — site is not live, no data to preserve

---

## 1. Goal & success criteria

Replace the MailerLite integration with Sender.net end-to-end. After this work:

- All MailerLite code, env vars, table names, KV prefixes, and docs references are gone from the repo (build artifacts and `.codesight/**` excepted — regenerated by tooling).
- `worker/src/lib/sender/` exports the same surface shape as today's `worker/src/lib/mailerlite/`, so handler/cron call sites change by import path + factory name only.
- `npm test` passes against the new client.
- A documented operator setup checklist (`docs/sender-setup.md`) lists every Sender dashboard object the code expects to exist (groups, custom fields, automations, trigger URLs).

Out of scope: data migration, vendor abstraction layer, transactional email wiring (`POST /v2/message/send` is not consumed by any handler today).

## 2. Constraints learned from Sender.net's API

| | MailerLite (today) | Sender.net (target) |
|---|---|---|
| Base URL | `https://connect.mailerlite.com` | `https://api.sender.net/v2` |
| Auth | `Authorization: Bearer <key>` | `Authorization: Bearer <token>` |
| Subscriber upsert | `POST /api/subscribers` | `POST /v2/subscribers` (natural upsert) |
| Get by email | `GET /api/subscribers/{email}` | `GET /v2/subscribers/{email_or_id}` |
| Update fields (no status flip) | `POST` with field-only body | `PATCH /v2/subscribers/{email_or_id}` |
| Account-level unsubscribe | `PUT /api/subscribers/{email}` status=unsubscribed | `PATCH /v2/subscribers/{email}` status=unsubscribed |
| List groups | paginated GET | `GET /v2/groups?page=&limit=` |
| Add to group | `POST /api/subscribers/{id}/groups/{groupId}` | `POST /v2/subscribers/groups/{group_id}` body `{ subscribers: [...] }` |
| Remove from group | `DELETE /api/subscribers/{id}/groups/{groupId}` | `DELETE /v2/subscribers/groups/{group_id}` body `{ subscribers: [...] }` |
| Run automation by id | `POST /api/automations/{id}/run` | **No analog.** Use per-region "API Call Is Made" trigger URL. |
| Idempotency-Key header | Sent (SHA-256 of email) | Not supported. Natural idempotency only. |
| Custom field payload key | `bbq_region` | `{$bbq_region}` (literal `{$key}` token) |
| Status enum | active / unsubscribed / unconfirmed / bounced / junk | active / unsubscribed / bounced / reported_spam / non_subscribed |
| Rate limit signal | undocumented; 429 retried | `429` + `Retry-After` header |
| Validation failure | 422 | 422 with Laravel-style `{ message, errors: { field: [...] } }` |
| Webhooks | dashboard-configured | dashboard-configured, **Standard plan+** only |

Two material design impacts:

1. **No "run automation by id" REST verb.** The Friday cron has to trigger each regional automation via its "API Call Is Made" trigger URL (one URL per region, configured in the Sender dashboard).
2. **No `Idempotency-Key` header.** Client-side dedup (the existing `idempotency_key UNIQUE` constraint on the retry table) does the work alone. Sender's POST-on-existing-subscriber natural upsert + DELETE-on-nonmember no-op make this safe.

## 3. Module surface

New tree: `worker/src/lib/sender/{client,groups,tags,retry,errors}.ts`.

### 3.1 `client.ts`

```ts
createSenderClient(opts: SenderClientOptions): SenderClient

interface SenderClient {
  subscribe(input: SubscribeInput): Promise<SubscribeResult>
  updateSubscriberFields(email: string, fields: Record<string, unknown>): Promise<{ id: string } | null>
  getSubscriberByEmail(email: string): Promise<{ id: string } | null>
  unsubscribe(input: UnsubscribeInput): Promise<void>
  listGroups(): Promise<SenderGroup[]>
  assignGroup(subscriberId: string, groupId: string): Promise<void>
  removeGroup(subscriberId: string, groupId: string): Promise<void>
  triggerWeeklyDigest(input: TriggerDigestInput): Promise<void>
}

interface TriggerDigestInput {
  triggerUrl: string         // per-region "API Call Is Made" trigger URL
  idempotencyTag: string     // <region>:<sendDate> — passed in body for automation to read
}
```

- Base URL: `https://api.sender.net/v2`
- Auth: `Authorization: Bearer ${SENDER_API_TOKEN}`
- Default timeout: 8s (unchanged)
- `Idempotency-Key` header removed.
- `hashKey()` export removed (no longer needed).
- 404 on `removeGroup` swallowed (unchanged behavior).
- `subscribe()` is a natural upsert; no separate create/update branching.
- `unsubscribe()` is `PATCH /v2/subscribers/{email}` with `{ status: "unsubscribed" }` (account-level). Per-group "soft unsubscribe" (portfolio semantics) is `removeGroup()` across the seven `pitmaster_*` groups, **not** the account-level unsubscribe — same separation as today.
- Field-key wrapping (`bbq_region` → `{$bbq_region}`) happens inside `client.subscribe()` / `client.updateSubscriberFields()`. Callers pass plain `bbq_*` keys.

### 3.2 `groups.ts`

Ports unchanged in shape:

- `ALL_GROUP_NAME = 'pitmaster_all'`
- `regionToGroupName(region)`, `bbqGroupNamesForSubscriber(region)`, `allBbqGroupNames()`
- `resolveGroupId(client, kv, name)` — KV cache key prefix changes to `sender_group_id:`. On cache miss, hydrate full portfolio (`pitmaster_*` namespace) via `listGroups()` and write each name→id to KV.
- `assignBbqGroups(client, kv, subscriberId, region)` and `removeBbqGroups(client, kv, subscriberId)` — identical semantics, different transport.

### 3.3 `tags.ts`

Public surface unchanged:

```ts
toBbqSubscriberFields(input: BbqTagInput): BbqSubscriberFields
```

Keys stay `bbq_zip`, `bbq_city`, `bbq_state`, `bbq_region`, `bbq_cut_pref`, `bbq_cooker_pref`, `bbq_timezone`, `bbq_signup_date`. The `{$...}` wrapping is an internal client concern.

Null/undefined fields are omitted (unchanged). Empty string sent through PATCH explicitly clears template conditionals (unchanged).

### 3.4 `retry.ts`

- Table name: `sender_retry`.
- `request_kind CHECK IN ('subscribe', 'unsubscribe', 'digest_trigger')` — `digest_trigger` replaces `send`.
- `MAX_ATTEMPTS = 10`, `MAX_BACKOFF_MS = 6h`, `PARK_DELAY_MS = 365d`, exponential 1m→6h cap — all unchanged.
- `enqueue(...)`, `drain(...)` signatures unchanged; SQL targets `sender_retry`.
- Drain filters `request_kind IN ('subscribe', 'unsubscribe')` — `digest_trigger` failures are not auto-retried (cron re-runs hourly Friday window).

### 3.5 `errors.ts`

```ts
class SenderError extends Error {
  requestKind: 'subscribe' | 'unsubscribe' | 'digest_trigger'
                | 'group_assign' | 'group_remove' | 'group_list' | 'field_update'
  kind: 'http_5xx' | 'http_4xx' | 'timeout' | 'malformed' | 'network'
  status?: number
  get shouldRetry(): boolean
}
```

`shouldRetry`:
- `http_5xx`, `timeout`, `network` → true
- `http_4xx` with status in `{408, 425, 429}` → true
- 422 → **false** (validation; will not succeed on retry — unknown `{$field_key}` or invalid email)
- `malformed` → false

PII redaction: `redactSecrets()` from `worker/src/lib/redact.ts` is applied to `SenderError.message` at retry-queue persistence time (same pattern as today), scrubbing `Authorization: Bearer …` and `token=…`. No changes to `redact.ts` needed beyond updating the contained vendor-name string if/where it appears.

## 6. Handler diffs (mechanical)

All four handlers keep their HTTP routes, request/response shapes, and HMAC token logic. Diffs:

- **`subscribe.ts`**: `createMailerLiteClient` → `createSenderClient`; imports from `lib/sender/{client,tags,groups,retry,errors}`; retry enqueue `request_kind='subscribe'` unchanged; idempotency key construction unchanged (client-side use only).
- **`unsubscribe.ts`**: Same import-path renames. `client.getSubscriberByEmail` unchanged. `removeBbqGroups` still walks seven groups. Account `status=active` preserved (portfolio semantic).
- **`preferences.ts`**: Same renames. `client.updateSubscriberFields` semantic preserved (PATCH; status untouched). Field payload `{$bbq_*}` wrapping handled inside the client.
- **`status.ts`**:
  - SQL: `mailerlite_retry` → `sender_retry`.
  - Response field rename: `mailerlite_retry_pending` → `esp_retry_pending`, `mailerlite_retry_parked` → `esp_retry_parked` (vendor-neutral wire names so a future ESP swap doesn't break operator dashboards).
  - No live ESP calls (unchanged).

## 7. Friday cron rebuild (`worker/src/crons/fridayEmail.ts`)

This is the only non-mechanical code change.

Today: walks `REGIONS`, for each region check local-time Friday 06:00, claim a `friday_campaign_log` row, call `client.triggerCampaign({ automationId, idempotencyTag })` → `POST /api/automations/{id}/run`.

After:

- Region→env map updated to read `SENDER_DIGEST_TRIGGER_URL_{REGION}` instead of `MAILERLITE_AUTOMATION_{REGION}_ID`.
- Missing-URL regions skipped with a warning event (unchanged).
- D1 claim happens **before** the HTTP call (unchanged) — so a non-2xx response or parse failure cannot double-send.
- `client.triggerWeeklyDigest({ triggerUrl, idempotencyTag })`:
  - `POST <triggerUrl>` with `Authorization: Bearer ${apiToken}` and body `{ "tag": "<region>:<sendDate>" }`.
  - 8s timeout.
  - Non-2xx → `SenderError` with `requestKind: 'digest_trigger'`.
- Audience filtering remains a dashboard concern (each automation filters on `pitmaster_<region>` group).
- Worker sends one trigger per region per Friday — no per-subscriber fan-out.

## 10. `docs/sender-setup.md` content outline

Operator-side setup the code depends on:

1. **API token** — Sender dashboard → Settings → API access tokens → Create. Stored as worker secret `SENDER_API_TOKEN`.
2. **Custom fields** — create eight fields under Subscribers → Fields, key column exactly: `bbq_zip`, `bbq_city`, `bbq_state`, `bbq_region`, `bbq_cut_pref`, `bbq_cooker_pref`, `bbq_timezone`, `bbq_signup_date` (the last as type Date). Code emits `{$bbq_*}` payload keys; field keys must match.
3. **Groups** — create seven groups by name: `pitmaster_all`, `pitmaster_northeast`, `pitmaster_southeast`, `pitmaster_midwest`, `pitmaster_south_central`, `pitmaster_mountain`, `pitmaster_pacific`. IDs auto-assigned, cached in KV on first lookup. If a group is renamed or recreated, run `wrangler kv:key delete --namespace WEATHER_KV sender_group_id:<name>` to invalidate.
4. **Automations** — six automations, one per region. Start trigger = "API Call Is Made". Audience filter = subscriber is in group `pitmaster_<region>`. Content = the weekly digest template. Copy each automation's trigger URL into the matching worker secret `SENDER_DIGEST_TRIGGER_URL_<REGION>`.
5. **Sending domain** — `mail.pitmaster.tools` via CNAME + SPF/DKIM/DMARC. `SENDER_FROM_EMAIL`, `SENDER_FROM_NAME`, `SENDER_REPLY_TO` optional (only consumed if `POST /v2/message/send` is wired in the future).
6. **Webhooks (optional)** — Sender's webhooks are Standard plan+. If/when wired, the handler verifies HMAC-SHA256 of the raw body against the per-webhook signing secret.

## 11. Open calls baked in (override if wrong)

- `/api/status` JSON field names → vendor-neutral `esp_retry_pending` / `esp_retry_parked` (not `sender_retry_*`).
- Client-side retry dedup key stays `SHA-256(<requestKind>:<email-lowercased>)`. Not sent as a header.
- 422 (Sender validation failure) is **non-retryable** in `shouldRetry`.
- Squash all 0001–0004 migrations into one new `0001_init.sql` (3a, not 3b).
- D1 table `friday_campaign_log` retained as-is (vendor-neutral name).
- The optional `SENDER_FROM_*` env vars are retained even though no handler consumes them today, for forward parity.

## 13. Non-goals

- No data migration from a MailerLite production account (none exists).
- No transactional email wiring (`POST /v2/message/send` not consumed by any handler).
- No vendor-neutral abstraction layer (Option B rejected as YAGNI).
- No webhook handler implementation (current "future" status preserved; documented as Standard-plan+ requirement).
- No changes to `subscribers`, `events`, or `friday_campaign_log` table shapes beyond what migration squashing produces.
- No frontend / static-HTML structural changes beyond replacing user-visible MailerLite mentions.

## 14. Risks

- **Operator setup drift.** Code expects exact field keys, group names, and per-region trigger URLs. Setup checklist in `sender-setup.md` is the contract. Pre-flight: cron skips regions whose trigger URL secret is missing — same defensive behavior as today.
- **Rate limit headroom.** Friday cron emits at most six trigger POSTs per Friday window; subscribe/unsubscribe burst is low. `429` is retryable; `Retry-After` honored. No throttling code added.
- **Field validation failures non-retryable.** A typo'd custom field key returns 422 and parks the retry row immediately. Status endpoint surfaces parked counts; operator must inspect and fix.
- **Group ID drift.** If an operator deletes and recreates a group in Sender, the KV cache will return a stale id. Invalidation is a manual `wrangler kv:key delete` (documented).
